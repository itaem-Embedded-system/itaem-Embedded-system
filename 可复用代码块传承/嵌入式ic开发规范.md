# IC 驱动开发规范

## 1. 核心理念与目标

本规范旨在确保 IC 驱动代码具备**高可移植性**、**高稳定性**和**可维护性**。

- **中间件（Middleware）可移植性优先**：驱动核心逻辑必须与硬件平台完全解耦，确保无需修改即可移植到不同 MCU。
- **运行期高鲁棒性**：系统必须具备自动检测故障并从错误中恢复的能力（如传感器掉线、数据异常）。
- **分层明确**：严格区分业务逻辑、驱动逻辑与硬件适配层。

***

## 2. 架构分层原则

### 2.1 分层职责

| 层级                   | 模块位置             | 职责描述                                 | 允许的依赖                     |
| :------------------- | :--------------- | :----------------------------------- | :------------------------ |
| **应用层 (App)**        | `app/`, `main.c` | 业务调度、数据处理、故障监测与恢复、调试输出 (VOFA/Printf) | Middleware, Platform, HAL |
| **中间件 (Middleware)** | `middleware/`    | **驱动核心逻辑**。处理寄存器协议、状态机、数据转换。         | **禁止依赖 HAL**，仅依赖标准 C 库    |
| **平台适配层 (Platform)** | `platform/`      | **硬件抽象**。实现具体的 I2C/SPI 读写、延时函数。      | HAL, BSP                  |

### 2.2 严格与灵活的边界

- **Middleware 层必须“洁癖”**：严禁包含 `main.h`、`stm32_hal.h` 等硬件相关头文件。严禁直接调用 `HAL_` 开头函数。
- **Platform 层允许“脏活”**：允许根据具体芯片实现细节（如 DMA、中断、模拟时序）。
- **App 层负责“兜底”**：负责处理中间件返回的错误，并执行重试或重启策略。

***

## 3. 强制性规范 (MUST)

### 3.1 可移植性规范

1. **上下文注入**：驱动必须通过结构体指针（`ctx`）传入读写函数和句柄，禁止在驱动内部定义全局硬件句柄。
2. **环境隔离**：中间件代码中不得出现特定平台的宏定义（如 `#ifdef STM32F1`），所有平台差异必须通过适配层屏蔽。

### 3.2 鲁棒性与容错规范

1. **参数强校验**：所有对外接口（API）必须检查指针是否为空、枚举值是否在合法范围内。
2. **超时机制**：所有涉及硬件等待（如 `WaitUntil`、`While`）的循环必须包含超时退出机制。超时时间必须使用宏定义。
3. **异常检测与恢复（Watchdog 机制）**：
   - **通信错误监测**：应用层必须统计连续通信失败次数。
   - **数据异常监测**：应用层必须检查数据合理性（如：是否全为 0，是否超出物理量程）。
   - **自动恢复**：当错误计数超过阈值时，必须触发**完整重初始化流程**（Reset -> Init），不能仅做简单的清零忽略。

### 3.3 代码安全规范

1. **读-改-写 (Read-Modify-Write)**：修改寄存器位段时，必须先读取当前值，修改目标位，再写回，严禁直接覆盖整个寄存器。
2. **宏化管理**：所有寄存器地址、配置字、魔数（Magic Number）、超时时间必须使用宏或枚举定义。

***

## 4. 推荐性规范 (SHOULD)

1. **非阻塞延时**：在应用层轮询（Loop）中，建议使用基于时间戳（Tick）的非阻塞调度，避免使用 `HAL_Delay` 阻塞 CPU。
2. **调试分离**：调试代码（如 VOFA+ 波形发送、Printf 日志）应置于 `main.c` 或独立调试模块中，**严禁**混入 `middleware` 驱动文件。
3. **错误码统一**：建议统一使用负值表示错误（如 `-1` 参数错误，`-2` 通信错误），`0` 表示成功。

***

## 5. 实现模板参考

### 5.1 驱动上下文定义 (Middleware)

```c
// middleware/xxx_driver.h
typedef struct {
    // 读写接口注入
    int32_t (*write_reg)(void *handle, uint8_t reg, const uint8_t *data, uint16_t len);
    int32_t (*read_reg)(void *handle, uint8_t reg, uint8_t *data, uint16_t len);
    // 硬件句柄（透传给读写接口）
    void *handle;
} xxx_ctx_t;
```

### 5.2 容错恢复逻辑 (App)

```c
// app/xxx_app.c
void XXX_Loop(void) {
    // 1. 获取数据
    int ret = xxx_read_data(&ctx, &data);
    
    // 2. 数据有效性检查 (非零检查, 范围检查)
    if (ret != 0 || is_data_invalid(&data)) {
        error_count++;
    } else {
        error_count = 0; // 成功则清除计数
    }

    // 3. 触发恢复机制
    if (error_count > MAX_ERROR_THRESHOLD) {
        xxx_reset(&ctx); // 硬件复位
        xxx_init(&ctx);  // 重新初始化
        error_count = 0;
    }
}
```

## 6. 评审自检清单 (Checklist)

在提交代码前，请对照以下清单自查：

- [ ] **依赖检查**：`middleware` 文件夹下的 `.c` 文件是否包含了 `main.h` 或 `stm32_hal.h`？（如果有，请删除）
- [ ] **参数校验**：`set_xxx` 函数是否检查了参数范围？
- [ ] **超时保护**：`while` 循环是否有超时计数器？
- [ ] **调试代码**：`middleware` 中是否残留了 `printf` 或串口发送代码？（请移至 `app` 或 `main`）
- [ ] **恢复机制**：拔掉传感器再插上，系统能否自动恢复数据输出？
- [ ] **数据校验**：传感器输出全 0 或乱码时，能否触发重置？
